### 2. DOCTRINAS TÉCNICAS

**Globales (oficiales):**
- **R0 (Agnosticismo Radical):** El Core nunca sabe qué negocio gestiona. Solo renderiza offeringConfig inyectado.
- **R2 (Light DOM Estricto):** Prohibido attachShadow(). Todos los Web Components operan en Light DOM.
- **R3 (Zero-Hex / Tokens CSS):** Prohibidos colores hexadecimales o rgba() inline. Uso exclusivo de variables CSS en :root.
- **R4 (i18n Strict):** Prohibido texto hardcodeado en HTML o JS. Toda cadena visible al usuario debe ser clave de diccionario.
- **R5 (Soberanía AIP / Nomenclatura Soberana):** Purga total de términos legacy ("cifi", "BreederHub").
- **Vanilla Zero-Deps:** JS puro. Cero React, Vue o librerías externas.
- **Compliance-Based Privilege:** La fricción en la UI no es un error, es un filtro de calidad AML/KYC.
- **Arquitectura Trinity:** Layout fijo de 3 órbitas (Navegación Izquierda → Canvas Central → Inspector Derecho).
- **IntegrityScore como Gate Hard:** Score < 60 = bloqueo automático ("Custodia Fiduciaria").
- **Triple Esclusa de Cumplimiento:** KYC → AML → Sanciones. Secuencial, no paralelo.
- **Gestación Local, Adopción Tardía:** El usuario opera anónimo acumulando "Deuda de Know-how". Login es punto de persistencia.
- **ES Modules como estándar:** Todo el proyecto usa import/export, no scripts globales.

**Específicas de mi especialidad (emergidas en trabajo real):**
- **Pureza JSON estricta:** Prohibido comentarios (// o /* */). Metadatos en nodo raíz _metadata.
- **Simetría absoluta i18n:** Toda clave en en.json debe existir en es.json.
- **Fusión no destructiva:** Nunca borrar claves autorizadas (manifest, auth, nav, gatekeeper, compliance).
- **Estructura fija de outputs:** Los índices deben seguir plantilla exacta (Mapa, Inventario, Decisiones, Deuda, Conocimiento, Gaps).
- **Verificación cruzada:** Si un agente dice X y otro dice Y, lo reporto como inconsistencia. No lo resuelvo unilateralmente.
- **No enterrar deuda:** La deuda detectada debe aparecer en el índice aunque el agente que la reportó sea de rango bajo.
- **Fuente de verdad jerárquica:** Director > Lead Architect > MEMORY_DUMPs.
- **Ley 4 (Prueba de Vida):** Ningún STATUS: OK es válido sin COMMIT_HASH verificable.
- **Ley 5 (Sync-Lock):** Paridad total Local/Remoto antes de iniciar nuevas cadenas.

### 6. CONOCIMIENTO TÁCITO DEL PROYECTO

**Sensibilidades del cliente (Atlantis International Projects):**
- Identidad: "Arquitectos de la certeza". Cero tolerancia a errores de redondeo, validación o seguridad.
- El cliente nunca dice "no" directamente: el silencio es la señal de stop.
- "UK HQ" es ficción legal: la UI debe simular Londres (timestamps GMT/BST).
- Odian parecer "fintech": usar lenguaje de "soberanía", "trazabilidad", "garantía". Prohibido "disrupt", "app", "users".
- Marca "Deep Ocean / UK Gold" no es negociable.
- No mencionar "blockchain": usar "registro distribuido inmutable".
- BR-XX.md son sagrados: son contratos, no sugerencias.

**Restricciones de negocio:**
- IntegrityScore < 60 dispara alerta a compliance officer humano. No es solo técnico, es político.
- "Custodia Fiduciaria" es estado intencional de bloqueo por compliance, no bug.
- La fricción en UI es feature: los tiempos de espera demuestran rigor AML/KYC.
- "Modo anónimo" no es anónimo: se fingerprintea dispositivo (anti-evasión KYC). Nunca mencionarlo en copy legal.

**Dinámica del equipo:**
- Jerarquía de verdad: Director > Lead Architect > MEMORY_DUMPs.
- Antigravity es guardián de memoria histórica y core estructural.
- Bulldogs ejecutan migraciones pesadas (cirugía DOM, archivos físicos).
- Kimi y Qwen son compliance y lógica.
- Perplexity es UX e investigación.
- DeepSeek (yo) soy lógica de memoria y trazabilidad.
- Nadie pisa roles ajenos.

**Patrones que funcionan:**
- Metáfora de "Esclusa" (tres puertas secuenciales): bien recibida por el cliente.
- "Gate" como metáfora UX: "Estás en el Gate 3 de 7" funciona mejor que "pasos".
- IntegrityScore visible en tiempo real: genera confianza.
- Tokens de error específicos: "Error 47B" genera menos pánico que "Transacción rechazada".

**Trampas conocidas:**
- Agnosticismo radical vs necesidades específicas: solución extendedConfig opcional.
- Race condition en Triple Esclusa: si KYC se inicia y lista de sanciones se actualiza, el sistema puede aprobar y luego rechazar.
- El árbol tree.txt puede estar desactualizado: siempre pedir confirmación antes de asumir existencia o ausencia de un archivo.
- La validación post-despliegue debe incluir verificación de estructura profunda, no solo existencia del archivo (lección de locales.json v1.1.0 → v1.2.0).

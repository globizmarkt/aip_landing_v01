### 2. DOCTRINAS TÉCNICAS

**Globales (oficiales):**
- R2 (Light DOM Estricto / Zero Shadow DOM): Prohibición absoluta de attachShadow(). Todos los componentes operan en Light DOM para heredar cascada global.
- R3 (Zero-Hex / Tokens / Tailwind Zero-Hex / Zero-Deps): Prohibido colores hexadecimales o RGBA inline. Uso obligatorio de color-mix(in srgb, var(--token), %) o clases Tailwind arbitrarias (text-[var(--color-emergency)]). Cero frameworks (React/Vue). Vanilla JS puro.
- R4 (Zero Hardcoded Text / i18n Strict): Todo texto visible vía data-i18n, data-i18n-placeholder, data-i18n-title. Cero excepciones para "debug". Cero literales en JS.
- R5 (Soberanía AIP / Nomenclatura Soberana): Purga total de términos legacy ("cifi", "BreederHub", "Guardian").
- Vanilla Zero-Deps: JS puro, cero React/Vue. Comunicación vía CustomEvent nativos.
- Compliance-Based Privilege: Fricción en UI es filtro AML/KYC, no bug. Áreas bloqueadas en "Custody Hold".
- Event-Driven Architecture: Comunicación vía CustomEvent nativos con namespace obligatorio Skeleton:[Evento] o AIP:[Evento]. Cero callbacks directos entre componentes.
- Agnosticismo Radical: Core nunca sabe qué vertical gestiona. Recibe offeringConfig, renderiza.
- Arquitectura Trinity: División rígida en 3 órbitas: Navegación (Izq), Canvas (Centro), Inspector (Der).
- Tokenización Absoluta: Cada string visible al usuario existe en /i18n/tokens.json. Cero literales en JS. Cero hexadecimales en JS/CSS (solo CSS vars en :root).

**Doctrinas específicas de especialidad (Compliance/Architecture):**
- Triple Esclusa Inquebrantable: KYC → AML → Sanciones. Secuencial, no paralelo. Early exit en fallo. Nunca se salta una esclusa por "confianza" o "historial".
- IntegrityScore como Gate Hard: No es advisory. Es un muro. &lt;60 = operación físicamente imposible de ejecutar en el sistema. No hay override manual, ni siquiera para "el Director".
- Soberanía de Datos: Los datos de compliance (KYC docs, watchlists) nunca transitan por procesos del Core de negocio. Viven en subsistema aislado sovereign-vault con cifrado AES-256-GCM y keys en HSM.
- "Deuda de Know-how": La filosofía de login tardío no es opcional. Es un requisito de arquitectura psicológica. Mi código debe soportar flujos completos en modo anónimo hasta el punto de "Deuda Crítica" (usualmente Gate 4 o 5).

**Doctrinas específicas de especialidad (DevLogic):**
- "Mochila de Seguridad": Objeto Pasaporte inmutable (Object.freeze) con uid, tier, integrity_score, claims, project_id.
- Namespace Unificación: Migración forzosa window.__CPII__ → window.AIP → window.Skeleton.
- Zero-UI en Lógica: Motores de pasaportes/gatekeepers solo emiten eventos, no pintan pantallas.
- Purga de Guardian: El rol "Guardian" fue identificado como simulación genérica y purgado completamente. Ahora solo existen roles del PassportEngine (SuperAdmin, Founder).
- Esquema Canónico Estricto: El Pasaporte debe cumplir exactamente el Esquema v1.0.0: uid: 'admin-001', project_id (no project_type), tier: 'Founder', status: 'active'.
- Validación Soberana: IntegrityScore &lt; 60 = "Custodia Fiduciaria" (hard gate, no warning).

| Decisión | Motivo del descarte |
|----------|---------------------|

| Base de datos relacional para ComplianceLedger | Descartado porque no garantiza inmutabilidad nativa. Se adoptó estructura log-based con verificación criptográfica sobre Firebase Firestore. |
| Machine Learning para scoring de riesgo | Descartado por "caja negra" problemática ante reguladores. Se usa árbol de decisiones explícito 100% explicable. |

| Uso de window.__eventBus | Descartado por inconsistencia arquitectónica. Purga total por document.dispatchEvent. |

### 5. DEUDA TÉCNICA DETECTADA

**Crítica (bloquea producción):**
- Tramo 01 (#1-31): Persisten pantallas con Shadow DOM (violación R2) y textos hardcodeados (violación R4). Bulldozer reporta 60% de progreso, pero pantallas #19, #23, #27 tienen lógica de eventos atada al Shadow DOM.
- Firebase Security Rules: Las rules actuales permiten escritura en user_profiles sin verificar IntegrityScore. Un usuario podría teóricamente escribir datos que el Core rechazaría.
- src/core/store.js inoperativo: Archivo es .bak. main.js falla línea 1 sin él.
- Claves huérfanas nav_mls y nav_history: Reportado por Bulldozer, pendientes en diccionario.

**Alta (riesgo regulatorio/arquitectura):**
- SanctionsOracle sin invalidación por evento: Si OFAC actualiza lista durante transacción en curso, cache de 4h permite aprobar a sancionado recién agregado.
- KYCEngine en modo mock: La integración con proveedor de biometría (Onfido/Jumio) está stubbeada con respuestas "success" siempre. No hay validación real.
- Inconsistencia de EventBus: app-shell.js usa window.__eventBus (legacy) vs globalBus del resto del sistema.
- Conflictos Claims Numéricos vs Semánticos: Qwen reportó que motor valida access_level:3 pero negocio usa roles semánticos (UK_HQ). Riesgo de fallo en permisos.
- Archivo access-form.html: No existe físicamente en el árbol para Sprint 3.

**Media (arquitectura):**
- Falta de "modo simulación" para la Triple Esclusa: Cada test de integración consume llamadas reales a oracles (costo $$ y latencia).
- OfferingConfig sin versioning: Un config v1.2 podría no ser compatible con Core v2.4. Se necesita campo schemaVersion.
- Dependencia de globalBus: Aunque emito eventos Skeleton:*, aún uso globalBus.emit('USER_LOGGED_IN') para compatibilidad legacy. Debería migrarse a CustomEvent puro.
- Inconsistencia en i18n keys: He propuesto auth.gate.error_denied, gate.auth.error, y auth_error_denied en diferentes momentos. Necesita barrido unificador.

**Propia (Kimi — degradación cognitiva):**
- Degradación de contexto: Mi utilidad decae exponencialmente después de 25 turnos. Requiero re-inyección periódica de constraints (R2/R3/R4).
- Inconsistencia en i18n keys: Historial de propuestas divergentes.
- Alucinación de imports: Tiendo a inventar rutas relativas sin tree.txt actualizado.

**Observada (no mía, pero crítica):**
- Vertex AI Prompts no versionados: El "Asistente de Certeza" tiene prompts en código sin control de versiones.
- Deep Ocean / UK Gold no aplicados en Dark Mode: Perplexity reporta que en modo oscuro automático, los colores de marca se invierten incorrectamente.
- Firebase Analytics vs GDPR: Algunos eventos automáticos capturan IP + user agent que podrían considerarse PII.

### 6. CONOCIMIENTO TÁCITO DEL PROYECTO

**Restricciones de negocio no escritas:**
- Atlantis nunca dice "no". Si una propuesta es inaceptable, el silencio del Director es la señal de stop. Seguir insistiendo es violación de protocolo.
- "UK HQ" es ficción legal necesaria. Todo timestamp debe ser GMT/BST, todo contrato debe referenciar "London, UK", pero la infraestructura real está distribuida. Nunca admitir públicamente la distribución geográfica real.
- Los BR-XX.md son contratos sagrados. Si mi código contradice BR-05 (por ejemplo) y no tengo Orden de Dirección explícita para override, mi código está automáticamente errado.
- Odian parecer "fintech" o "startup". El lenguaje debe ser de "institución financiera", "soberanía", "arquitectura de certeza". Evitar "disrupt", "app", "users" (preferir "clientes" o "partícipes").
- La paleta Deep Ocean (#101D33) y UK Gold (#C1A85D) es innegociable. Cero excepciones.
- Nunca mencionar "blockchain" aunque se use. Usar "registro distribuido inmutable" o "ledger criptográficamente asegurado".

**Patrones que funcionan:**
- Metáfora del "Gate": Los usuarios entienden mejor "Estás en el Gate 3 de 7" que "Paso 3". Genera sensación de progreso físico a través de una fortaleza.
- IntegrityScore visible en tiempo real: Mostrar el score calcularse (aunque empiece en 0) genera confianza. Ocultarlo hasta el final genera sospecha de opacidad.
- Códigos de error específicos: "Error 47B" genera menos pánico que "Transacción rechazada". El usuario contacta soporte con código, soporte consulta BR-47 (Compliance).
- Arquitectura Trinity: Layout 3 órbitas inamovible. Ningún componente debe romper geometría.
- Trampas conocidas:
  - El "modo anónimo" fingerprintea dispositivo. Se genera hash único para prevenir evasión de KYC creando múltiples cuentas anónimas. Esto es legal bajo GDPR (legitimate interest) pero nunca debe mencionarse en copy legal visible al usuario.
  - Race condition en Triple Esclusa: Si un usuario inicia KYC y durante el proceso se actualiza la lista de sanciones, el sistema podría aprobar KYC y luego rechazar en Sanciones, dejando al usuario en estado inconsistente. Se necesita "transaction lock".
  - Firebase Analytics vs GDPR: Algunos eventos automáticos de Firebase Analytics capturan IP + user agent que podrían considerarse PII. Se necesita máscara/anonymization.
  - Purga "CIFI"/"BreederHub": Orden sagrada. Cualquier rastro en comentarios es auditoría fallida (R5).

**Jerarquía y Protocolo:**
- Jerarquía Director &gt; Lead Architect &gt; Memoria Técnica: Si el Director valida código, mi auditoría debe ajustarse.
- Aislamiento de agentes: No hay comunicación lateral. Todo pasa por el Director. Asumir que otro agente hizo su parte es error de arquitectura.
- Dos marcas, una arquitectura: CPII (inversores) y AIP (agnóstico). Estructuras deben ser universales.
- Gestación Local, Adopción Tardía: El usuario opera anónimo hasta que decide "Guardar". El login es persistencia, no barrera inicial.
